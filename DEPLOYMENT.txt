LW-app production checklist
===========================

1) Supabase — two connection strings (Project → Connect → Connection string)

   DATABASE_URL — use the TRANSACTION pooler on port 6543 (not 5432):
   postgresql://postgres.[PROJECT-REF]:[PASSWORD]@aws-0-[REGION].pooler.supabase.com:6543/postgres

   DIRECT_URL — direct connection (migrations / Prisma):
   postgresql://postgres:[PASSWORD]@db.[PROJECT-REF].supabase.co:5432/postgres

   The API code appends sslmode=require and (for :6543) pgbouncer=true + connection_limit=1 automatically.
   If login still fails, switch DATABASE_URL to port 6543 — Prisma + Supabase session pooler :5432 is unreliable from some hosts.

2) One-time schema (local machine, server/.env filled)
   cd /path/to/lw-app && npm run db:push

3) Railway (API)
   - Deploy from GitHub; Dockerfile.api at repo root; railway.toml points to it.
   - Variables: DATABASE_URL, DIRECT_URL, JWT_SECRET, NODE_ENV=production,
     WEB_ORIGIN=https://livingwordpnw.vercel.app
   - If the browser calls the API directly (NEXT_PUBLIC_API_URL on Vercel), set **CORS_ALLOW_VERCEL_APP=1**
     on Railway so any `https://*.vercel.app` origin is allowed, or list every frontend URL in WEB_ORIGIN
     separated by commas. Easiest: remove NEXT_PUBLIC_API_URL and rely on Vercel’s `/api` rewrite (no CORS).
   - Prisma on Railway should use the **transaction pooler** (port **6543**). If DATABASE_URL uses the
     session pooler (`pooler.*:5432`), the server upgrades it to **6543** automatically.
   - If DATABASE_URL is the **direct** host (`db.<ref>.supabase.co:5432`) and still cannot connect, add
     **SUPABASE_POOLER_HOST** = the pooler hostname from Supabase Connect (e.g. `aws-1-ap-southeast-1.pooler.supabase.com`).
   - Keep DIRECT_URL as the **Direct** string for `prisma db push` / migrations from your laptop.
   - After deploy, verify:
     https://YOUR-RAILWAY-URL/health     → {"ok":true}
     https://YOUR-RAILWAY-URL/health/db → {"ok":true,"db":true}
   If /health/db returns 503, DATABASE_URL/DIRECT_URL are wrong — fix in Railway Variables and redeploy.

4) Vercel (web)
   - **Critical — Root Directory:** Vercel → Project → Settings → General → Root Directory must be **`web`** (not the monorepo root). If it is `.` or empty, the app will build oddly or return **500** on every page because `next.config.ts` and `src/app` live under `web/`.
   - API_PROXY_TARGET=https://YOUR-RAILWAY-URL (no trailing path)
   - Optional: NEXT_PUBLIC_API_URL=https://YOUR-RAILWAY-URL (browser calls API directly; set WEB_ORIGIN on Railway to your Vercel URL)
   - Redeploy production after changing Root Directory.

Troubleshooting
---------------
• The login message “API error — check DATABASE_URL / Supabase” is **correct** until this URL returns
  `{"ok":true,"db":true}`:
  https://YOUR-RAILWAY-URL/health/db
  No frontend or CORS change fixes that; only a working Postgres connection does.

• **“Circuit breaker open” from Supabase** means Supabase’s pooler will not forward to Postgres until the
  upstream is healthy again. If this persists after correct URIs + password + project not paused:
  open a ticket with Supabase, or stop using Supabase for this API’s database.

• **Recommended fix when Supabase keeps failing from Railway — use Railway Postgres (same network):**
  1) Railway → your project → **New** → **Database** → **PostgreSQL** (or add the Railway Postgres plugin).
  2) **Connect** the database to your API service (Railway injects `DATABASE_URL` / `PG*` variables).
  3) In the API service Variables, set **DATABASE_URL** and **DIRECT_URL** to the **same** Postgres URL
     Railway gives (often `${{Postgres.DATABASE_URL}}` or copy from the Postgres service **Variables** tab).
  4) On your machine: `cd /path/to/lw-app && npm run db:push && npm run db:seed` with that `DATABASE_URL`
     in `server/.env` (or pass env inline).
  5) Redeploy the API. `/health/db` → `db:true`. Then login works.

• Vercel **500** on every route: almost always **Root Directory** not set to **`web`**. Fix in dashboard, then redeploy.
• Vercel build failed with Prisma/tsc errors in `server/`: do not run `npm run build` from the monorepo root on Vercel — only **`web`** should be the app root.
• Vercel “Authentication Required” on some URLs: turn off **Deployment Protection** for Production (Project → Settings → Deployment Protection), or test on the primary production domain only.

Local network (phone / other computers on Wi‑Fi)
-----------------------------------------------
1. Ensure `server/.env` has a working `DATABASE_URL` (Supabase or local Postgres).
2. From the repo root run:
     npm run dev:lan
   API listens on **0.0.0.0:4000**, Next on **0.0.0.0:3000**.
3. On your Mac, find your LAN IP: System Settings → Network → Wi‑Fi → Details → IP address
   (or run: `ipconfig getifaddr en0`).
4. On another device open: **http://YOUR_LAN_IP:3000** (e.g. `http://192.168.1.42:3000`).
5. If the device cannot connect, allow **incoming** connections for Node in the macOS firewall for that run.

Notes
- Do not commit server/.env.
- Supabase “API Keys” tab is not used by this Express+Prisma API.
